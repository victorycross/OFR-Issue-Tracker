US Quick Suite consists of several components, an SDK (formerly known as QuickCreate), CICD Pipelines (formerly known as QuickDeploy) and a hosting platform within US Technology & NGC CaaS. The SDK provides development teams with a prompt based (CLI) guided process of generating the right components for an application.  It will then create components such as: Microservices (NodeJS/NestJS or .NET), a Web Application (Angular), an Orchestrator, Functional Tests, Pipeline Scripts (EasyDevOps), Charts, Flux manifests, Terraform scripts, etc.  Then, using the CICD pipeline, it can deploy code to either the US Tech Kubernetes clusters or the NGC CaaS clusters, thus expediting project code creation and delivery to different environments.
This document will focus on the Quick Suite platform's hosting infrastructure. Quick Suite utilizes managed (PaaS) Kubernetes clusters for container orchestration, specifically on AKS (Azure Kubernetes Service) and GKE (Google Kubernetes Engine) in the US Tech environment and AKS clusters in NGC CaaS (Next Gen Cloud - Container as a Service).
NOTE: The infrastructure environment is split among (2) CIs.  This was how the original environment was deployed and when we tried to combine the two CI records, we were told this was not feasible due to ServiceNow limitations.

## 1.3 Platform Technology Overview

1.) US Tech clusters - (PaaS) consists of Sandbox and Main AKS & GKS clusters.  There are (3) Main AKS clusters (Non-Prod, Prod, and a dedicated Rancher instance) and (2) Main GKE clusters (Non-Prod & Prod).  The Sandbox environment mimics the main environment but to a smaller scale and is intended for internal testing with no active user facing workloads.  All clusters are managed and administered by Quick Suite’s Release Engineering Team (US TCS App Services).  The AKS clusters are hosted within PwC’s hybrid cloud solution and the GKE clusters have a VPN connection back to PwC’s internal network managed by NGC Ops.
2.) NGC CaaS clusters - (PaaS) consists of (3) pairs of AKS clusters, one for each environment (Dev, Stage, Production).  The primary cluster in each pair is hosted in Azure East and the second cluster is hosted in Azure West, also serving as the secondary DR cluster.  They are all managed and administered by NGC Ops with the Release Engineering Team having an elevated level of access to the environment, but only within the context needed to support Quick Suite applications deployed into the environment.
3.) Rancher (SUSE) - is a tool used to simplify the management of multiple Kubernetes clusters by providing a single pane of glass for the clusters and is deployed on a dedicated AKS environment.  There are (2) Rancher environments for US Tech Sandbox & Main clusters as well as (3) different Rancher instances for NGC CaaS Dev & Stage and Prod clusters.

Rancher provides user management and RBAC, security policy management, capacity management and backup and recovery features for Rancher itself.  Application teams are granted access to specific Rancher Projects (typically representing an application) and manage a collection of namespaces hosted on a Kubernetes cluster.  These may represent dev or test environments as well any other supporting workloads that may be better suited in a different namespace (depending on business requirements).

Rancher is deployed on a dedicated AKS instance. PwC users authenticate via Azure AD (@pwc.com account). They must also be provisioned access to specific projects in Rancher via Release Engineering Team.  Projects are logical groupings of namespaces that represent a PwC application. Application teams only have access to their specific deployments in Rancher projects.  Within their specific project, applications teams have access to their namespaces, ingress rules, deployments, services and container logs.
4.) Azure Traffic Manager (SaaS) is used within the US Tech & NGC CaaS AKS clusters for internal DNS resolution.
- Hostnames for applications hosted on the US Tech AKS clusters are configured to resolve to a Traffic Manager which resolves to the AKS clusters Ingress controller
- GKE Clusters do NOT have a traffic manager involved, hostnames resolve directly to the Ingress controller on the respective clusters.
- Applications hosted on the NGC CaaS environment leverage individual traffic managers to resolve to the Ingress controllers on clusters in Azure East and Azure West within NGC CaaS. This allows applications to determine if the traffic should flow to either cluster, but by default,  the traffic manager on the Azure East cluster is the primary cluster.
5.) Wildcard DNS - Quick Suite utilizes a Wildcard DNS and each environment has a wildcard DNS name resolving to its respective Nginx ingress controller. The Wildcard DNS is intended for applications hosted for internal *.qd.pwcinternal.com based applications.

US Tech GKE Clusters - the wildcard DNS entries resolve directly to the Nginx ingress controller of the respective cluster.

US Tech AKS Clusters – uses an Azure Traffic Manager where all the application hostnames resolve to the traffic manager which then defines the proper IP address of the Ingress Controller.  This was originally intended to provide DR failover capabilities from one Azure region to another (e.g. East → West).  At the time, AKS did not provide multi-zonal support and nodes were limited to one Azure zone within a region.  Since then, AKS does now support multiple Availability zones and the US Tech clusters adopted this model, and subsequently kept the Traffic Manager in place for flexibility in updating ingress controller IPs as needed.

NGC CaaS AKS Clusters – individual applications integrated with NGC CaaS have their own traffic manager that controls the east/west traffic.  There is a common traffic manager for the environment but this is intended to cover individual applications that opted out of the individual traffic manager.  Each Azure region has their own wildcard DNS names.

6.) Nginx Ingress Controllers - Each Quick Suite Kubernetes cluster has one or more Nginx Ingress Controllers deployed to it and running within the cluster to manage traffic coming into the cluster.

For US Tech AKS/GKE clusters:
- Dev/Test environments normally reside within the Non-Prod cluster
- Stage/Production environments reside within the Prod cluster.
- Additional environments may be used by application teams (e.g. uat, perf, etc.), but would leverage either the Non-Prod or Prod cluster based on suitability.

AKS
- HTTP Nginx Ingress Controller for HTTP/HTTPS (:80/:443) traffic
- TCP Nginx Ingress Controller for Non-HTTPS traffic used by Neo4J/Kafka/RabbitMQ apps
- XLOS Apigee Nginx Ingress Controller for dedicated Apigee traffic

GKE
- HTTP Nginx Ingress Controller for HTTP/HTTPS (:80/:443) traffic
- TCP Nginx Ingress Controller for non-HTTP/HTTPS traffic used by Neo4J/Kafka/RabbitMQ apps

For NGC CaaS AKS clusters
- Dev/Test environments normally reside within the Dev cluster
- Stage environments reside within the Stage cluster.
- Production environments reside within the Prod cluster.
- Additional environments may be used by application teams (e.g. uat, perf, etc.), but would leverage any of the CaaS clusters based on suitability.
- HTTP Nginx Ingress Controller for HTTP/HTTPS (:80/:443) traffic

It is an internal-facing cache for the US Tech Quick Suite clusters (AKS & GKE) & NGC CaaS (AKS) clusters. The cache is used only by applications that require its usage due to an NIS vulnerability issue identified for that application.

Azure Redis Cache is used to address the PEN testing vulnerability identified below:

- Improper Logout:  JWT Token can be reused after a user logs out of the Web App. The PEN Testers used a man in the middle attack to intercept and/or record the traffic.  This was reported in several Quick Suite Applications.
This vulnerability deals with user sessions not being invalidated properly in OpenAM. The QuickSuite solution will create a unique key (UUID) representing the user’s login session and stores it in the redis cache for each user upon login. As the user logs into an application, a QSLogin microservice is called to create/store the UUID value into the Redis cache. If a user logs out of the application, the QSLogout microservice is called and it clears/removes the UUID value from the session cache. If an attempt to login is made and the UUID is already present in the session store, then it will reject the login attempt.

- Non-Production clusters share a common Non-Prod Redis Cache
- Production clusters share a common Production Redis Cache
- Cross environment communication will not be allowed and is blocked via firewall

We’ve utilized Azure Service Bus for efficient internal communication between these microservices.

## 3.1 Authentication & Access

The US Quick Suite platform leverages Azure AD and PwC Identity for internal user authentication, providing secure access to the Rancher management interface and Quick Suite applications. Privileged users and administrators authenticate via PwC Identity with MFA. While external user access is currently not provided, any future integration will adhere to PwC’s security policies regarding registering/onboarding with Imperva. Service accounts and service principals are managed through Azure AD, ensuring role-based access control aligned with application requirements.
The Quick Suite SDK provides a common authentication library that leverages the PwC Identity solution.  The specific authentication method (cert, MFA, etc) is determined by the individual applications’ data classification and business requirements.  This is normally determined by the entityID submitted by the application team during their onboarding process with PwC Identity directly.

## 3.6 Security Stack


### 1. Access to K8S Clusters

- Azure Active Directory (AAD): Used for authenticating users (*@admin.pwc.com & *@gcp.pwc.com) and controlling access to the US Tech AKS & GKE cluster. Integrates with Kubernetes Role-Based Access Control (RBAC).
- Azure AD Groups: Manages access to Azure resources, including AKS, using role assignments.
- GCP Groups: Manages access to Azure resources, including AKS, using role assignments.

### 2. Network Security

- Network Policies: Calico network policy defines rules limiting traffic between pods in all the clusters.
- Azure Network Security Group (NSG): AKS clusters have NSGs in place that are managed by PwC’s Network/Perimeter team.
- Azure Virtual Network (VNet): Segregates and isolates the AKS cluster network. Subnets, Network Security Groups (NSGs) are used for fine-grained control.
- Google Cloud Virtual Private Cloud (VPC) & Firewall - used to isolate network environments with GCP and limit access to approved endpoints and are administered by PwC’s NGC Operations team.
- Palo Alto Firewall: AKS clusters sit behind the Palo Alto firewall managed by the PwC Perimeter team.
- ExpressRoute (Azure) & VPN (GCP) : Provide secure connectivity between on-premises networks and Azure & Google Cloud.
- HTTPS & TLS Certificates: Access to applications require HTTPS and application webapps are configured with TLS certificates

### 3. Data Security

- Azure Key Vault: Leveraged to store credentials used by the Release Engineering team.
- Encryption: Uses encryption for data at rest and in transit. Encryption at rest with a platform managed key is configured for the AKS/GKE clusters.

### 4. Container Security

- Container Image Scanning: Leverages PwC’s Twistlock (and JFrog X-Ray) to scan container images for vulnerabilities.
- PwC Private Container Registry: Leverages PwC’s JFrog Artifactory as the container registry.  Artifactory access uses PwC Identity for access to container images.
- Quick Suite SECT Team: Dedicated team that works with application teams on vulnerabilities identified by Twistlock

### 5. Cluster Security

- Node Security: Ensure nodes are securely configured and regularly updated.  Release Engineering does not access the nodes directly
- OS and Kubernetes Patching: Kubernetes on the control plane and the data plane is upgraded in accordance with the support windows provided by Microsoft and GCP.

## 3.8 Third Party Software & Services


Rancher
Rancher (SUSE Labs) is an open source tool that simplifies the management of Kubernetes clusters by providing a single pane of glass in our cross vendor clusters.  For the US Tech clusters, Release Engineering manages and administers the Rancher instance and for NGC CaaS clusters, NGC Ops manages and administers a separate Rancher instance.

It solves three primary issues for Kubernetes users:

- It is a multi-cluster management tool for administering Kubernetes clusters. Whether it is an existing Kubernetes cluster, or hosted kubernetes cluster like GKE, EKS or AKS, Rancher can centrally manage all of these clusters. Specifically, it addresses user management and RBAC, security policy management, capacity management, delegated administration, Rancher cluster backup and recovery, etc.
- It provides a UI and an API and for users to interface with all their managed Kubernetes clusters. Native Kubernetes tools are also accessible to users via Rancher. This layer provides integrated tooling that makes using Kubernetes quite easy.
- All of this is delivered as a single platform, so you can move from version to version of Rancher and upgrade managed clusters independently.

Rancher is deployed in a dedicated AKS instance. PwC users authenticate via Azure AD (@pwc.com account). They must also be provisioned access to specific projects in Rancher.  Projects are logical groupings of namespaces that represent a PwC application.

Application teams have access to their specific deployments through Rancher projects.  Within their specific project, applications teams have access to their namespaces, deployment, and pod logs.

Nginx Ingress
Each Quick Suite Kubernetes cluster utilizes one or more Nginx Ingress Controllers (https://artifacthub.io/packages/helm/ingress-nginx/ingress-nginx) for all the respective environments contained within that cluster.  This applies to both US Tech and NGC CaaS clusters.

## 3.9 Disaster Recovery, Business Continuity & Incident Response Processes

Disaster recovery for critical applications and critical services must be planned and documented. For more information please refer to this site: Service Portfolio - Disaster Recovery/Business Continuity (DR/BC) (FY23).  Here is a  Disaster Recovery Plan Template to help with your planning efforts.
A key component of a territory’s data protection program is to have organizational and technical measures in place to secure personal data it processes from the risk of accidental or unlawful destruction, loss, alteration, or unauthorised disclosure or access. Every application team will need to onboard to the PwC Monitoring and Alerting Service to facilitate incident response actions.
US Tech Environment

The AKS / GKE clusters were built with Availability Zones enabled within the respective vendors (Microsoft/Google). Within a cloud region there are multiple availability zones located within an Azure or GCP Region, Availability Zones allow a cluster to place a node within one zone (which can consist of one or more physical data centers). Each region consists of multiple zones, so within say the Azure East region, the AKS cluster uses (3) availability zones, meaning each node (vm machine) that is part of the Kubernetes cluster is staggered across the 3 different availability zones within Azure East.

If there's 30 nodes and one zone experienced a failure, roughly 1/3 of the total nodes sitting in that one zone would become unavailable.  Kubernetes would scale additional nodes on the other zones (auto-scale is enabled on the clusters) and Kubernetes scheduler would re-schedule deployments (pods) to the available new nodes.  Depending on the pods configured for a particular application, there may be a slight outage window while the pods are rescheduled.  As a best practice for Production environments, each workload should have (2) or more pods configured for each of their deployments.  If the app is utilizing more than 1 pod for your app, the corresponding kubernetes service would load balance the requests and no outages should occur.
US Tech AKS / GKE clusters are provisioned in a single cloud region with multiple availability zones.  AKS is in Azure East with (3) availability zones (zone1, zone2, zone3).  GKE is in GCP US-East1 (us-east1-b, us-east1-c, us-east1-d).  If the entire region becomes unavailable for a particular cloud provider, then the cluster sitting in that region would also become unavailable.
NGC CaaS Environment
Quick Suite’s CICD Pipelines are configured to deploy workloads to both Azure East & West if the application is hosted on NGC CaaS and the application requires DR capabilities, based on library variables set within ADO for the CICD pipeline.
NGC CaaS AKS clusters are provisioned in two cloud regions (Azure East & Azure West).  Within Azure East, which is the primary region in which all applications are deployed, NGC is in the process (June  through Aug 2024 update window) of rebuilding the node pools to leverage (3) availability zones.  Azure.  Azure West serves as the Disaster Recovery (DR) zone for Quick Suite applications deployed to NGC CaaS.  Apps are normally deployed as a warm or hot standby in Azure West.
The NGC CaaS AKS clusters deployed in Azure West only leverage one availability zone (Azure West does not support multiple availability zones).  As the need arises within NGC CaaS, the cluster will be rebuilt in one of the other Azure West Regions that do support multiple availability zones at a later time.

# Appendix A - ACQ Information

NOTE: This section is for application/solution team situational awareness to understand the background process of assessing the application for risks and compliance with firm standards.
Application Readiness Reviews (ARR) are a mandatory security review service that the Application Readiness team conducts for all tools or technologies hosted in the production environment or any technology service hosted in a non-production environment with access to production data or production data systems (i.e. via API).
The Application Compliance Questionnaire (ACQ) is completed as part of the ARR process. The ACQ enables the uniform capture of Application Readiness assessment results by NIS in Archer for compliance scoring.
Project teams navigating through the ARR process will be asked to provide architectural design documentation showcasing security controls (Security Design Document), which will be used as the AR team completes the ACQ and other ARR processes. By completing the Security Design Document in advance of requesting ARR, project teams can reduce the overall cycle time to release.
